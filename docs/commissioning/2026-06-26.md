## Axis analog outputs (AOUT) — root cause found and fixed ✅
X produced ±V but Y/Z/W read 0 V at TB3. Not damage, not the 7I85, not the analog
supply. **Cause: on the 7I97 the four axis PWM channels share ONE group enable —
`pwmgen.00.enable` is the master for channels 0–3.** Manual (SERVO ENABLE OUTPUTS):
*"ENA0 through ENA3 are switched in common; ENA4 and ENA5 can be independently
switched."* Mesa confirmed: *"PWM enable 0 enables channels 0..3 — for channel 1 to
work you must have PWM enable 0 AND PWM enable 1 set true."*

Why only the test scripts hit it: they forced the channel's own `pwmgen.0N.enable`
but left `pwmgen.00.enable` linked to `sig-x-amp-enable` (= FALSE, machine off).
`setp` can't override a pin tied to a signal → group stayed gated → Y/Z/W = 0 V.
X worked only because the X case unlinks ch0. Fix: unlink + force `pwmgen.00.enable`
for EVERY axis, relink on cleanup. Y then metered ~+5 V at TB3-8. All 4 axes output.
Recorded: memory `ned-7i97-pwm-enable0-gates-group`. In normal running this is
automatic (joints enable together → sig-x-amp-enable high → group live).

## Drives reconnected; open-loop motion per axis ✅
Built `tools/drive_move.sh` — explicit flags `-volts Y [-time X] [-counts Z]`, stops
on whichever limit hits first, streams the axis encoder, enables all drives, forces
one channel, disables + relinks on exit. Confirmed each drive moves under command
and the encoder tracks. Note: count-stop resolution is limited by the 0.25 s bash
poll (at 1 V the axis flies ~2300 counts/poll), so `-counts` is coarse — a tight
count stop would need a HAL comparator.

## Axis directions logged (`docs/commissioning/axis_directions.md`)
+voltage → physical motion, established empirically:
- **Y:** +V → STARBOARD, encoder.01 counts up (+1 V → +2276 counts/0.25 s).
- **X:** +V → AFT (gantry aft), counts up.
- **Z:** +V → spindle UP.

## Gantry linked (`tools/drive_gantry.sh`) ✅
X (pwmgen.00) and W (pwmgen.03) driven from ONE shared signal (`net sig-gantry-vel
→ both value pins`, `sig-gantry-en → both enables`) so both get the identical
command on the same servo cycle — sequential `setp`s would skew and rack the frame.
Basis: a HAL signal fans out to many reader pins (one writer, many readers);
mirrors LinuxCNC's own `gantry` component ("drives multiple motors from a single
axis input"). Ran `-volts 0.1`: **X and W track, same direction, no rack — motors
are NOT mirror-mounted.**

## W encoder was DEAD → FIXED (detached connector) ✅
During the linked move W moved but `encoder.03.rawcounts` stayed flat 0. Long
debugging arc — initially MIS-diagnosed as a firmware mapping problem (thought the
`7i85sd` bitfile read ch3 off the DB25). **That was wrong.** Re-read the live IDROM
+ ran a per-channel hand-test (`tools/watch_encoders.sh`): the encoder map is
STRAIGHT, axis N = encoder N, straight off TB1/TB2. The `MuxedQCount 3`/`4` lines on
the DB25 in `readhmid` are the **7I85's counter instances, NOT `encoder.03`/`.04`**
(misreading the Chan column was the whole error). **Real cause: W's X4 (DB15)
feedback connector was completely detached** at the cabinet. Reseated → e3 reads,
hand-turning W counts on encoder.03. No reflash needed; firmware is correct.

## Gantry open-loop tracking — skew is expected
Over ~20 k counts the X↔W skew is ~500 counts (**2.5 %**). This is the OPEN-LOOP
velocity-trim mismatch between two independently-trimmed SDSM analog drives at the
same command voltage — no position feedback in the loop to correct it; it
integrates over the move. Worst at low command (offset is a bigger fraction).
Closed-loop PID per joint (normal running) kills it. Added `skew/X` and `skew/W`
percent columns to `drive_gantry.sh` to watch whether it holds steady (trim) or
grows (worse).
